## Purpose
Fulfil the module contracts for ISS-002 and ISS-004 by adding all missing fields to `SettingsStore` (2.3) and wiring every module that currently falls back to a local default to read from the shared store instead.

## Success Condition
Every file tagged `// ISS-002` or `// ISS-004` reads its value from an injected `SettingsStore`; `TerminalProfile.default` is no longer constructed locally in `TerminalView`; opening macOS Settings shows a Terminal tab (font, scrollback, colours) and an expanded Editor tab (debounce intervals, locale, max file size); changing a terminal-font-size field in Settings is reflected by `TerminalView` on next session open.

## Steps

- [ ] 1. **Extract TerminalColor to Foundation 2.3**
   What: Create `2 Foundation/2.3 Settings/TerminalColor.swift` — move the `TerminalColor` struct verbatim from `7 Terminal/TerminalProfile.swift` (keep the static palette constants). Remove the definition (not the usage) from `TerminalProfile.swift`.
   Why: `SettingsStore` lives in Foundation and must not import the Terminal module (SR-1). `TerminalColor` is a settings-layer primitive with no terminal-specific behaviour — it belongs alongside `EditorFont` and `AppTheme`.

- [ ] 2. **Add terminal fields to SettingsStore**
   What: In `SettingsStore.swift` add five new stored properties — `terminalFontName: String` (default `"Menlo"`), `terminalFontSize: Double` (default `13.0`), `terminalScrollbackLimit: Int` (default `5_000`), `terminalForeground: TerminalColor` (default `TerminalColor(red:0.88,green:0.88,blue:0.88)`), `terminalBackground: TerminalColor` (default `TerminalColor(red:0.08,green:0.08,blue:0.09)`). Add corresponding `DefaultsKey` string constants, paired `setTerminal…(_:)` mutators (each persists via `PersistenceService`), and `loadFromDefaults()` branches. Follow the existing `autoSaveEnabled` pattern exactly.
   Why: Resolves ISS-002 — these are the fields the Terminal module guide documents as "sourced from Settings (2.3)" but that `SettingsStore` has never owned.

- [ ] 3. **Add editor-settings fields to SettingsStore**
   What: Add seven more stored properties — `editorMaxFileSizeBytes: Int` (default `10_485_760`), `markdownDebounceInterval: TimeInterval` (default `0.3`), `asciiDebounceInterval: TimeInterval` (default `0.3`), `htmlDebounceInterval: TimeInterval` (default `0.3`), `spellCheckDebounceInterval: TimeInterval` (default `0.5`), `asciiTriggerKey: String` (default `"_"`), `spellCheckLocale: String?` (default `nil`). Add paired `DefaultsKey` constants, mutators, and `loadFromDefaults()` branches.
   Why: Resolves ISS-004 — these values are currently hardcoded in six editor files, creating silent per-file divergence and blocking testability.

- [ ] 4. **Wire Terminal to SettingsStore (ISS-002)**
   What: In `TerminalView.swift` add `@Environment(SettingsStore.self) private var settings`; replace any `TerminalProfile.default` construction with `TerminalProfile(fontName: settings.terminalFontName, fontSize: settings.terminalFontSize, foreground: settings.terminalForeground, background: settings.terminalBackground, scrollbackLineLimit: settings.terminalScrollbackLimit)`. Grep `TerminalManager.swift` for any `TerminalProfile.default` usage and apply the same substitution. Remove all `// ISS-002` comments.
   Why: Without this wiring the new `SettingsStore` fields have no consumers and the terminal remains on its local default forever.

- [ ] 5. **Wire editor sub-modules to SettingsStore (ISS-004)**
   What: In each of the six files tagged `// ISS-004`, replace the local constant with a `SettingsStore` read. Non-view types that cannot use `@Environment` should accept a `SettingsStore` parameter in their `init` (pass it from the owning SwiftUI view, which already has `@Environment(SettingsStore.self)`):
   - `EncodingGuard.swift` → `settings.editorMaxFileSizeBytes`
   - `MarkdownLanguageProvider.swift` → `settings.markdownDebounceInterval`
   - `ASCIIArtLanguageProvider.swift` → `settings.asciiDebounceInterval`
   - `HTMLLanguageProvider.swift` → `settings.htmlDebounceInterval`
   - `SpellingGrammarChecker.swift` → `settings.spellCheckLocale` (pass to `NSSpellChecker`)
   - `QuickfixPresenter.swift` → `settings.spellCheckLocale` (same `language:` argument)
   Remove all `// ISS-004` comments once replaced.
   Why: Closes the local-default debt that ISS-004 documents.

- [ ] 6. **Build real SettingsView with Terminal and Editor tabs**
   What: In `SputnikApp.swift`, replace the placeholder `SettingsView` with a `TabView` containing four tabs: **Appearance** (existing `theme`, `editorFont`), **Editor** (existing `autoSaveEnabled`, `lineNumbersEnabled`, `wordWrapEnabled`, plus the new `editorMaxFileSizeBytes`, debounce intervals, `asciiTriggerKey`), **Spelling & Grammar** (existing `spellCheckEnabled`, `grammarCheckEnabled`, plus new `spellCheckLocale`), and **Terminal** (new `terminalFontName`, `terminalFontSize`, `terminalScrollbackLimit`, foreground/background colour pickers). Use `@Bindable` to drive each control from `SettingsStore`'s `set…` mutators.
   Why: Without a UI, users can never change the new settings; the data model is complete but invisible. The placeholder comment in `SputnikApp.swift` explicitly defers this to "module 2.3 UI pass" — this step delivers it.

- [ ] 7. **Update Module Guide 2.3**
   What: In `1 Setup/Module Guides/2 Foundation/2.3 Settings/guide.md`, add all twelve new fields to the **State owned** section; update `last_updated: 2026-06-08`.
   Why: The guide is the source of truth for the module — it must stay in sync with the implementation (per working conventions in CLAUDE.md).

## Risks and Constraints
- **SR-1 (module boundary):** Foundation must not import Terminal. After extracting `TerminalColor`, run a grep to confirm no `import` statement pulls in the Terminal module from Foundation files.
- **`TerminalColor` move:** The struct moves modules but the name is unchanged — `TerminalProfile.swift` needs no code changes beyond removing the duplicated definition. If the app uses a single Swift module (not split packages), no import statement changes are needed at all.
- **`@Observable` is additive:** Adding new stored properties to `SettingsStore` is non-breaking; existing observers keep working. No migration or version bump needed.
- **`@MainActor` threading:** `SettingsStore` is `@MainActor`. All six editor sub-modules also mutate `NSTextStorage` and call `NSSpellChecker` on `@MainActor` — the new reads are safe on the main thread.
- **Settings UI colours:** macOS doesn't ship a standard `ColorPicker` in Settings scenes. Use `ColorWell` via `NSColorWell` wrapped in `NSViewRepresentable`, or SwiftUI's `ColorPicker` (available since macOS 11).
