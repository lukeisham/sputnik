## Purpose
Create `SputnikCommands` ‚Äî the missing SwiftUI `Commands` struct that wires all six menu bar menus (Sputnik, File, Edit, View, Window, Help) to `AppState` actions, attaches it to `SputnikApp`, and unblocks the Help menu trigger that module 9 (Resources) depends on to present help panels.

## Success Condition
The app compiles. Every menu item listed in the 2.0 guide's Menu Bar spec is present, uses the correct keyboard shortcut, and calls the correct `AppState` method or system action. Opening `Help > Markdown Help` sets `AppState.requestedHelpTopic = .markdown` (observable by module 9). The Settings‚Ä¶ menu item opens the Settings window.

## Steps

- [ ] 1. **Create `SputnikCommands.swift` in `2 Foundation/2.0 App Overview/`**
   What: Create the file with a `SputnikCommands` struct conforming to SwiftUI `Commands`. This struct wires all six menus using SwiftUI's declarative `CommandMenu`, `CommandGroup`, and system groups (`importFromDevices`, `help`, `newItem`, `printItem`, `saveItem`, `textEditing`, `undoRedo`, `windowList`, `windowSize`). `AppState` is read via `@Environment`. The struct goes in `2.0 App Overview/` because it is a cross-cutting assembly type that references every menu ‚Äî it is the architectural glue between the menu bar spec and the runtime state, matching the 2.0 guide's role as the app overview.
   Why: No menu bar code exists today. The 2.0 guide specifies every menu item with its label, keyboard shortcut, and behaviour. `SputnikCommands` is the single source of truth for wiring them. The Help menu items (`Markdown Help`, `HTML Help`, `ASCII Art Help`, `Grammar Help`) each set `appState.requestedHelpTopic` to the corresponding `HelpTopic` case ‚Äî this is the trigger that module 9's plan step 3 depends on.

- [ ] 2. **Wire the Sputnik menu**
   What: Add `CommandGroup(replacing: .appInfo)` with About Sputnik (system default), a divider, `Button("Settings‚Ä¶")` with `‚åò,` shortcut that opens the Settings scene, a divider, Hide Sputnik (`‚åòH`), Hide Others (`‚å•‚åòH`), Show All, a divider, and Quit Sputnik (`‚åòQ`).
   Why: The Sputnik menu is the first menu per the guide. The Settings‚Ä¶ button triggers the Settings scene already defined in `SputnikApp` via `Settings { SettingsView() }`. The Hide/Quit items use system-provided actions.

- [ ] 3. **Wire the File menu**
   What: Add `CommandGroup(replacing: .newItem)` for New Tab (`‚åòT`) ‚Üí `appState.newUntitledDocument()`. Add a custom `CommandMenu("File")` containing: New Window (`‚áß‚åòN`), a divider, Open‚Ä¶ (`‚åòO`) triggering an `NSOpenPanel`, Open Recent submenu (reading `appState.recentFiles` + Clear Menu action ‚Üí `appState.clearRecentFiles()`), a divider, Close Tab (`‚åòW`) ‚Üí close the active document via `appState.closeDocument(activeID)`, Close Window (`‚áß‚åòW`), a divider, Save (`‚åòS`), Save As‚Ä¶ (`‚áß‚åòS`), a divider, Print‚Ä¶ (`‚åòP`). Replace the system `saveItem` and `printItem` groups.
   Why: The Open Recent submenu dynamically reads `appState.recentFiles` (newest first) and calls `appState.openDocument(url:)` on selection. This is the primary document-lifecycle menu.

- [ ] 4. **Wire the Edit menu**
   What: Use `CommandGroup(replacing: .undoRedo)` for Undo (`‚åòZ`)/Redo (`‚áß‚åòZ`) (system-provided). Replace `.textEditing` for Cut/Copy/Paste/Select All (system-provided). Add a custom Find submenu: Find‚Ä¶ (`‚åòF`), Find and Replace (`‚å•‚åòF`), Find Next (`‚åòG`), Find Previous (`‚áß‚åòG`). Add a Spelling and Grammar submenu: Check Now (`‚åò;`), Check While Typing (toggle bound to `settingsStore.spellCheckEnabled`), Show Corrections.
   Why: Most Edit items are system-provided and require no custom code beyond replacing the command group. The Spelling and Grammar submenu reads/writes `SettingsStore` toggles and is the user-facing control for the spell/grammar checker (module 3.5).

- [ ] 5. **Wire the View menu**
   What: Add a custom `CommandMenu("View")` with: Toggle File Tree (`‚å•‚åò1`) ‚Üí `appState.toggleVisibility(.left)`, Toggle Preview (`‚å•‚åò2`) ‚Üí `appState.toggleVisibility(.centerLower)`, Toggle Right Panel (`‚å•‚åò3`) ‚Üí `appState.toggleVisibility(.right)`, Toggle Terminal (`‚å•‚åò4`) ‚Üí `appState.toggleTerminal()`, a divider, Focus: Editor (`‚åÉ‚åòE`) ‚Üí hide left + right, show center, Focus: Reader (`‚åÉ‚åòR`) ‚Üí hide left + editor, show preview full-width, Restore Default Layout (`‚åÉ‚åò0`) ‚Üí `appState.restoreDefaultLayout()`, a divider, Appearance submenu with Light Mode / Dark Mode / Use System Setting ‚Üí `settingsStore.setTheme(.light)` / `.dark` / `.system`.
   Why: The View menu is the most complex ‚Äî it toggles panel visibility, switches focus modes, and controls appearance. Focus modes are implemented as convenience presets that call the same `toggleVisibility` methods in sequence. The Appearance submenu reads/writes `SettingsStore.theme`.

- [ ] 6. **Wire the Window menu**
   What: Use `CommandGroup(replacing: .windowSize)` for Minimize (`‚åòM`) and Zoom (system-provided). Add custom items: Move Tab to New Window, Merge All Windows (stubs for multi-window support). Use `CommandGroup(replacing: .windowList)` for the open-documents list (Swift auto-populates from `WindowGroup`).
   Why: The Window menu's open-documents list is automatically populated by SwiftUI's `WindowGroup` when documents are opened. Move Tab to New Window and Merge All Windows are stubs ‚Äî multi-window support is a future feature.

- [ ] 7. **Wire the Help menu**
   What: Add `CommandGroup(replacing: .help)` with: Sputnik Help (`‚åò?`) ‚Üí `appState.requestedHelpTopic = .sputnik`, a divider, Markdown Help ‚Üí `appState.requestedHelpTopic = .markdown`, HTML Help ‚Üí `appState.requestedHelpTopic = .html`, ASCII Art Help ‚Üí `appState.requestedHelpTopic = .asciiArt`, Grammar Help ‚Üí `appState.requestedHelpTopic = .grammar`, a divider, Release Notes (stub), Report an Issue‚Ä¶ (stub ‚Äî opens default mail client with prefilled subject).
   Why: This is the critical dependency for module 9. Each help menu item sets `AppState.requestedHelpTopic`, which module 9's help panels observe to show/hide themselves. Without this step, the entire Resources plan's step 3 (panel visibility routing) has no trigger and no help panel can ever appear.

- [ ] 8. **Attach `SputnikCommands` to `SputnikApp`**
   What: In `SputnikApp.swift`, add `.commands { SputnikCommands() }` to the `WindowGroup` scene. Inject `AppState` via `.environment(appState)` is already done ‚Äî `SputnikCommands` reads it via `@Environment(AppState.self)`.
   Why: SwiftUI's `Commands` must be attached to a `Scene`. `WindowGroup` is the correct attachment point. Without this line, `SputnikCommands` compiles but never renders ‚Äî the menu bar stays empty.

- [ ] 9. **Validation ‚Äî verify every menu item**
   What: Build and run. Open each menu and verify: (a) every item from the 2.0 guide's menu bar spec is present with the correct label and shortcut; (b) `Help > Markdown Help` sets `requestedHelpTopic` (verify by adding a temporary `onChange` print in `ContentView`); (c) `View > Toggle Terminal` hides/shows the terminal placeholder; (d) `Sputnik > Settings‚Ä¶` opens the Settings window; (e) `File > Open Recent` shows recent files (if any); (f) `View > Appearance` submenu switches themes.
   Why: Menu bars are notoriously easy to get subtly wrong ‚Äî wrong shortcut, missing divider, incorrect `CommandGroup` replacement that duplicates or removes system items. A systematic verification against the guide's spec catches these before module 9 starts depending on the Help menu.

## Risks and Constraints

- **No `NSOpenPanel` or `NSSavePanel` integration yet**: The File > Open‚Ä¶ and Save As‚Ä¶ items need `NSOpenPanel`/`NSSavePanel`. A lightweight `NSOpenPanel` wrapper is sufficient for Open‚Ä¶ ‚Äî the chosen file URL is passed to `appState.openDocument(url:)`. Save As‚Ä¶ is a stub since the save pipeline isn't built yet (module 3 owns file I/O).

- **Focus modes are convenience presets**: Focus: Editor and Focus: Reader are implemented as sequenced calls to existing `toggleVisibility` methods rather than new `AppState` methods. This avoids adding new state to `AppState` and keeps the logic in the presentation layer. If focus modes need to be persisted in the future, they can be promoted to `LayoutState` then.

- **`CommandGroup` replacement conflicts**: Replacing `CommandGroup` removes system-provided items. For example, replacing `.newItem` removes the system's "New Window" item. The plan explicitly re-adds every item from the guide's spec. Items not in the spec that the system normally provides (e.g. "Enter Full Screen" in the View menu) will be removed ‚Äî this is intentional per the spec.

- **Move Tab to New Window / Merge All Windows are stubs**: Multi-window support is not in scope. These items are present in the menu per the spec but are disabled for now.

- **Foundation-only change**: This plan touches only Foundation files (`2 Foundation/`). No other module is affected. This is a pure Foundation assembly task (SR-1).
